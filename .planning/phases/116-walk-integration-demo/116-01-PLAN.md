  <action>
Rewrite `prepare_walk_data` to use `build_move_table` and `make_combined_predicate` instead of `get_legal_moves_and_oracle`. This is the data-layer rewrite.

**prepare_walk_data changes:**
1. Remove `get_legal_moves_and_oracle` import, add imports for `build_move_table` from chess_encoding, `make_combined_predicate` from chess_predicates, `_KNIGHT_OFFSETS`/`_KING_OFFSETS` from chess_encoding.
2. For each level, determine side-to-move (white if even, black if odd).
3. For white levels: call `build_move_table([('wk', 'king'), ('wn', 'knight')])` -> d_max=16, branch_width=ceil(log2(16))=4.
   For black levels: call `build_move_table([('bk', 'king')])` -> d_max=8, branch_width=3.
4. Per user decision: independent tables per side with independent branch widths.
5. Build one `make_combined_predicate` per table entry (not per level). Each entry `(piece_id, dr, df)` gets its own predicate parameterized by:
   - `piece_type`: 'king' if piece_id in ('wk','bk'), 'knight' if piece_id == 'wn'
   - `dr, df`: from the table entry
   - `board_rows=8, board_cols=8`
   - `enemy_attacks`: White moves -> `[(_KING_OFFSETS, 'bk')]`; Black moves -> `[(_KING_OFFSETS, 'wk'), (_KNIGHT_OFFSETS, 'wn')]`
   - `n_friendly`: White moves -> 1 (one other friendly piece qarray besides the moving piece); Black -> 0 (no other black pieces in KNK)
   - `n_enemy`: White -> 1; Black -> 2
6. Store per-entry qarray key mappings for argument resolution at evaluate_children time:
   - For white king moves: piece_key='wk', friendly_keys=['wn'], king_key='wk', enemy_keys=['bk']
   - For white knight moves: piece_key='wn', friendly_keys=['wk'], king_key='wk', enemy_keys=['bk']
   - For black king moves: piece_key='bk', friendly_keys=[], king_key='bk', enemy_keys=['wk', 'wn']
7. Build offset-based oracle per level using a new `_make_apply_move_from_table` factory (defined in chess_walk.py, NOT chess_encoding.py). The oracle:
   - Takes `(wk_arr, bk_arr, wn_arr, branch)` same as existing `_make_apply_move`
   - For each branch value i -> move_table[i] = (piece_id, dr, df)
   - Maps piece_id to board index: {'wk': 0, 'bk': 1, 'wn': 2}
   - For each valid (src_r, src_f) -> (dst_r, dst_f) = (src_r+dr, src_f+df) where 0 <= dst coords < 8
   - Inside `branch == i` condition, flip src and dst squares
   - Off-board destinations are skipped classically (no gates emitted) -> identity for invalid moves
   - Use @ql.compile(inverse=True)
8. Per-level move_data dict structure:
   ```python
   {
       'move_table': [(piece_id, dr, df), ...],
       'move_count': len(move_table),
       'branch_width': ceil(log2(move_count)),
       'apply_move': compiled_oracle,
       'predicates': [compiled_pred_0, compiled_pred_1, ...],
       'entry_qarray_keys': [
           {'piece': 'wk', 'friendly': ['wn'], 'king': 'wk', 'enemy': ['bk']},
           ...
       ],
   }
   ```

**n_friendly calculation for white moves:** When white king moves, the moving piece IS the king, so friendly_keys=['wn'] (1 friendly). When white knight moves, friendly_keys=['wk'] (1 friendly). So n_friendly=1 for all white entries. For black, n_friendly=0.

**IMPORTANT:** The `friendly_keys` list should NOT include the moving piece itself. The piece_qarray (first arg to predicate) is the moving piece. friendly_qarrays are the OTHER friendly pieces that could be at the destination.

**walk_step signature note:** The function signature and internal callers (evaluate_children, apply_diffusion, etc.) need to accept `move_data_per_level` which now contains predicates and entry_qarray_keys in addition to the existing keys.
