**Investigation approach (from scratch per user decision):**

1. First, verify BUG-01 fix is in place (MAXLAYERINSEQUENCE=300000 from Plan 01).

2. Re-enable the skipped qarray multiplication test at the SMALLEST possible scale:
   - Single element qarray, width=1, multiply by 1 (identity). Does this crash?
   - Single element qarray, width=2, multiply by 2. Does this crash?
   - Incrementally increase to find the minimal crash reproduction.

3. Check if single `qint.__imul__` works outside qarray context:
   ```python
   c = ql.circuit()
   a = ql.qint(3, width=4)
   a *= 2
   ```
   If this crashes too, the bug is in `qint.__imul__`, not in qarray iteration.

4. If single qint __imul__ works but qarray doesn't, the bug is in the iteration/GC interaction in `_inplace_binary_op`.

**Root cause candidates (investigate in order):**

A. **allocated_qubits flag not swapped in __imul__** (lines 695-728 of qint_arithmetic.pxi):
   - `__imul__` swaps `self.qubits` and `result.qubits`, swaps `allocated_start`, copies `bits`
   - But does NOT swap `allocated_qubits` flag
   - When `result_qint` goes out of scope and gets GC'd, `_do_uncompute` or `__del__` may try to free qubits that `self` now owns
   - Fix: After the qubit swap, set `result_qint.allocated_qubits = False` and/or `result_qint._is_uncomputed = True` to prevent GC cleanup of the swapped qubits

B. **GC-triggered uncomputation during iteration:**
   - When iterating qarray elements, creating intermediate results for each element may trigger GC
   - GC may call `__del__` on stale intermediates from previous iterations
   - Fix: Suppress GC during iteration or mark intermediates properly

C. **Width change breaking array consistency:**
   - `self.bits = result_qint.bits` could change width if multiplication produces wider result
   - Fix: Ensure result width matches original element width

**Apply the fix, then rebuild:** `pip install -e .`
