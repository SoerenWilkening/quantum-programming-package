<objective>
Fix BUG-04: Mixed-width QFT addition produces incorrect results when operand widths differ (e.g., 3-bit qint + 5-bit qint).

Purpose: The `__add__` operator creates a result of `max(self.bits, other.bits)` width and calls `hot_path_add_qq` which dispatches to `QQ_add(result_bits)`. The sequence `QQ_add(result_bits)` expects `result_bits` qubits for BOTH target and source registers, but when `other_bits < result_bits`, the qubit array has uninitialized positions for the missing source qubits.

Output: Correct mixed-width QFT addition for all width combinations up to 8 bits, zero regressions on same-width addition.
</objective>

  <action>
**Root cause:** In `hot_path_add_qq` (hot_path_add.c), when `self_bits != other_bits`, the qubit array layout passes `other_bits` qubits starting at position `self_bits`, but `QQ_add(result_bits)` expects `result_bits` qubits for the source register at positions `[result_bits..2*result_bits-1]`.

**Fix approach — zero-extend in `__add__` (qint_arithmetic.pxi):**

The cleanest fix is in the Python-level `__add__` operator. Currently it does:
```python
a = qint(width=result_width)
a ^= self     # Copy self bits
a += other    # Calls addition_inplace(other)
```

When `self.bits < result_width`, `a ^= self` already correctly zero-extends (upper bits stay 0). But `a += other` calls `addition_inplace(other)` which passes `other.bits` to C. The C side then uses `QQ_add(result_bits)` but the qubit array only has `other_bits` entries for the source register.

**Fix in `__add__` (qint_arithmetic.pxi):** Before calling `a += other`, zero-extend `other` to `result_width` if needed:
```python
if type(other) == qint and (<qint>other).bits < result_width:
    padded_other = qint(width=result_width)
    padded_other ^= other  # Zero-extend via XOR copy
    a += padded_other
else:
    a += other
```

Apply the same fix to `__radd__` (which has identical logic).

Also apply the same fix to `__sub__` and `__rsub__` since they have the same mixed-width pattern (`a ^= self; a -= other`).

**Do NOT touch:** `addition_inplace`, `hot_path_add_qq`, or `IntegerAddition.c`. The C-level `QQ_add` is correct for same-width operands. The fix should ensure both operands always have `result_width` by the time they reach the C layer.

**Important:** Only apply the zero-extension when both operands are `qint` (the `type(other) == qint` check already exists in `__add__` at the `result_width` computation). The CQ path (`type(other) == int`) does not have this issue because `CQ_add` only requires the target register width.
