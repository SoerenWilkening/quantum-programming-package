  <name>Task 1: Create benchmark_eval.py with EVAL-01/02/03 measurements and BENCH-04 report</name>
  <files>scripts/benchmark_eval.py</files>
  <action>
Create `scripts/benchmark_eval.py` -- the evaluation and report generation script.

The script has two modes: (1) run EVAL measurements (subprocess-based, like Plan 01), (2) generate BENCH-04 comparison report from all collected data.

**Section 1 -- EVAL-01: Multiplication Evaluation**
- Measure QQ_mul first-call generation cost at widths 1-16 (already in bench_raw.json from Plan 01)
- Additionally measure: CQ_mul per-call cost (NOT cached -- each call regenerates). Use subprocess, 5 iterations per width, widths 1-16.
- Compare multiplication generation cost vs addition generation cost at same widths (read addition data from bench_raw.json)
- Compute the ratio: `mul_cost / add_cost` per width to show how much more expensive multiplication generation is
- Produce recommendation: "hardcode" if mul generation cost is >10x addition AND the binary size impact is acceptable, "skip" otherwise. Document the reasoning.

**Section 2 -- EVAL-02: Bitwise Evaluation**
- Q_xor, Q_and, Q_or first-call costs are already in bench_raw.json from Plan 01
- Extract and analyze: compare bitwise generation cost vs addition generation cost
- Compute generation cost in absolute terms (microseconds) and relative to addition
- Produce recommendation: "skip" if bitwise generation cost is under 100us for all widths (expected based on O(N) trivial generation), "hardcode" only if surprisingly expensive

**Section 3 -- EVAL-03: Division Evaluation**
- Division is Python-level, NOT a C sequence. Measure end-to-end cost of `a // b` (floor division) at widths 1-8 (keep small -- division creates many sub-operations)
- Use subprocess per measurement (5 iterations per width). For classical divisor: `a // 3`. For quantum divisor: `a // b` (may be very slow for large widths due to exponential loop).
- Document that division cost is in the Python loop structure (N comparisons+subtractions for classical, 2^N for quantum), not in sequence generation
- Produce recommendation: "skip" -- cost is fundamentally not in sequence generation, hardcoding individual sequences would not help

**Section 4 -- BENCH-04: Comparison Report Generation**
- Load `benchmarks/results/bench_raw.json` (from Plan 01)
- Combine with EVAL measurements
- Generate `benchmarks/results/benchmark_report.json` with structure:
```json
{
  "import_time": {"with_hardcoded_ms": ..., "so_sizes_bytes": {...}},
  "first_call_generation_us": {
    "QQ_add": {"1": ..., ..., "16": ...},
    ...all 9 operations...
  },
  "cached_dispatch_ns": {
    "QQ_add": {"8_hardcoded": ..., "17_dynamic": ...},
    "CQ_add": {"8_hardcoded": ..., "17_dynamic": ...}
  },
  "evaluation": {
    "multiplication": {"cost_vs_addition_ratio": {...}, "recommendation": "...", "reasoning": "..."},
    "bitwise": {"max_cost_us": ..., "recommendation": "skip", "reasoning": "..."},
    "division": {"classical_cost_us": {...}, "recommendation": "skip", "reasoning": "..."}
  },
  "amortization": {
    "import_overhead_ms": ...,
    "avg_generation_saving_per_call_us": ...,
    "break_even_calls": ...,
    "analysis": "..."
  }
}
```
- Generate `benchmarks/results/benchmark_report.md` with:
  - Title: "Phase 62: Hardcoded Sequence Benchmark Report"
  - Section 1: Import Time Analysis (BENCH-01 data + .so size analysis)
  - Section 2: First-Call Generation Cost Table (BENCH-02, all 9 ops x 16 widths, formatted as markdown table)
  - Section 3: Cached Dispatch Overhead (BENCH-03, comparison table)
  - Section 4: Amortization Analysis (formula: `break_even = import_overhead_ns / generation_saving_per_call_ns`; explain what this means in practical terms)
  - Section 5: Multiplication Evaluation (EVAL-01, with cost table and recommendation)
  - Section 6: Bitwise Evaluation (EVAL-02, with cost table and recommendation)
  - Section 7: Division Evaluation (EVAL-03, with cost explanation and recommendation)
  - Section 8: Summary Recommendations (one-line recommendation per operation category)

**CLI interface:**
- `--eval-only` to run EVAL measurements without regenerating the report
- `--report-only` to generate the report from existing data without re-measuring
- Default: run EVALs then generate report
- `--widths` to override width range for division (default "1-8" since division is slow)

**IMPORTANT considerations from research:**
- CQ_mul is NOT cached (returns fresh malloc each call) -- measure it differently from QQ_mul
- Division at width > 8 with quantum divisor may take very long -- set a per-subprocess timeout of 30s and skip widths that timeout
- Division uses `__floordiv__` for `//` and there is a `__mod__` for `%` -- only measure `//`
- The amortization formula: if hardcoded sequences add X ms to import but save Y us per first-call, break-even = X*1000/Y calls
