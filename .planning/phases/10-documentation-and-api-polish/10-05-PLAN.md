  <files>python-backend/quantum_language.pyx</files>
  <action>
Debug the qint_mod * qint_mod segfault issue.

The problem is in qint_mod.__mul__ (around line 2264-2300). When called with another qint_mod:
1. qint_mod.__mul__ calls qint.__mul__(self, other)
2. qint.__mul__ creates result = qint(width=result_width) - a plain qint, not qint_mod
3. qint.__mul__ calls self.multiplication_inplace(other, result)
4. multiplication_inplace at line 802 checks `isinstance(other, qint)` which returns True for qint_mod
5. It then calls QQ_mul(result_bits) which appears to be producing garbage/segfault

Investigation steps:
1. Test if qint * qint works correctly (basic case)
2. If basic qint * qint fails, the issue is in C-layer QQ_mul or qubit array setup
3. If basic qint * qint works but qint_mod * qint_mod fails, issue is in type handling

Based on testing, qint * qint produces garbage memory (not a crash but wrong values), suggesting the C-layer QQ_mul output is not being read back correctly OR the qubit array setup in multiplication_inplace is wrong.

Potential fixes to investigate:
- Check that qubit_array is properly sized for the width
- Verify the result qint is properly initialized before multiplication
- Check if NUMANCILLY is sufficient for the operation
- Ensure qubits are being extracted correctly from the 64-element arrays

If the C-layer fix is too complex, consider a Python-level workaround that:
- Uses repeated addition: a * b = a + a + ... (b times)
- OR uses shift-and-add multiplication algorithm
- Document as temporary fix until C-layer is corrected

For now, add a check in qint_mod.__mul__ that detects when other is qint_mod and raises NotImplementedError with a clear message, then update README to show only qint_mod * int examples.

Alternative: If you can identify the bug in multiplication_inplace or the C layer, fix it properly.
