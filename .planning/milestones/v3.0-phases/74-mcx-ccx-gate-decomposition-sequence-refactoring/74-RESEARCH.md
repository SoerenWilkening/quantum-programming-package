# Phase 74: MCX/CCX Gate Decomposition & Sequence Refactoring - Research

**Researched:** 2026-02-17
**Domain:** Quantum gate decomposition (MCX -> CCX, CCX -> Clifford+T) and C file refactoring
**Confidence:** HIGH

## Summary

This phase has three main workstreams: (1) decomposing all MCX gates (3+ controls) into CCX/CX/X gates via the AND-ancilla pattern already proven in ToffoliMultiplication.c, (2) adding an opt-in `toffoli_decompose` option that decomposes CCX gates into the Clifford+T basis {H, T, Tdg, CX}, and (3) refactoring the largest C files for maintainability.

The MCX decomposition has a proven reference implementation in Phase 72-03 (`emit_cMAJ_decomposed` and `emit_cUMA_decomposed` in ToffoliMultiplication.c). The pattern is: for MCX(target, [c1, c2, ext_ctrl]), replace with CCX(and_anc, c1, c2) + CCX(target, and_anc, ext_ctrl) + CCX(and_anc, c1, c2). This extends recursively for higher control counts. The exact same pattern needs to be applied to all other MCX emission points in the codebase.

The CCX -> Clifford+T decomposition is the standard textbook decomposition (Nielsen & Chuang, Fig. 4.9 / Shende & Markov 2008): 2 H gates, 4 T gates, 3 Tdg gates, and 6 CX (CNOT) gates. This totals 7 T/Tdg gates per CCX, which is proven optimal for exact (non-relative-phase) decomposition. Three new gate enum values (T, Tdg -- H already exists) need to be added to `Standardgate_t` in types.h.

The file refactoring targets ToffoliAddition.c (1968 lines) as the primary pain point. Secondary targets are hot_path_add.c (529 lines) and ToffoliMultiplication.c (579 lines). The refactoring must be a pure behavior-preserving split before any decomposition work begins.

**Primary recommendation:** Execute as three sequential workstreams: (A) pure refactoring first, (B) MCX auto-decomposition second, (C) CCX->Clifford+T third. Each workstream is independently testable.

### Recursive AND-Ancilla Decomposition for Arbitrary Control Counts

For MCX with k controls (k >= 3), the recursive pattern is:

```
MCX(target, [c1, c2, ..., ck])
  -->
If k == 3:  (base case, proven in Phase 72-03)
  CCX(and_anc, c1, c2)          -- compute partial AND
  CCX(target, and_anc, c3)      -- apply via ancilla
  CCX(and_anc, c1, c2)          -- uncompute AND
  Cost: 3 CCX, 1 ancilla

If k > 3:  (recursive case)
  CCX(and_anc_1, c1, c2)        -- compute AND of first 2
  MCX(target, [and_anc_1, c3, ..., ck])  -- recurse with k-1 controls
  CCX(and_anc_1, c1, c2)        -- uncompute AND
  Cost: 2 CCX + MCX(k-1), 1 ancilla per recursion level
  Total ancilla: k - 2
  Total CCX: 2*(k-2) + 1 = 2k - 3
```

For the codebase, the relevant MCX control counts are:
- **3 controls** (most common): cMAJ/cUMA, controlled mul width-1, cQQ_add_bk CCX->MCX
- **n controls** (comparison): CQ_equal_width for n >= 3
- **n+1 controls** (comparison): cCQ_equal_width for n >= 3, cQQ_add_bk MCX->MCX

### Standard CCX -> Clifford+T Decomposition

The exact (non-relative-phase) decomposition of CCX(target=t, ctrl1=c1, ctrl2=c2) into {H, T, Tdg, CX}:

```
    c1: ─────────────●──────────────●────●──T────●─
    c2: ──────●──────┼───────●──────┼──T─X──Tdg──X─
     t: ──H───X──Tdg─X──T───X──Tdg─X──T─H─────────
```

Gate sequence (reading left to right):
```
H(t)
CX(t, c2)       -- CNOT: target=t, control=c2
Tdg(t)
CX(t, c1)       -- CNOT: target=t, control=c1
T(t)
CX(t, c2)       -- CNOT: target=t, control=c2
Tdg(t)
CX(t, c1)       -- CNOT: target=t, control=c1
T(t)
T(c2)
H(t)
CX(c2, c1)      -- CNOT: target=c2, control=c1
T(c1)
Tdg(c2)
CX(c2, c1)      -- CNOT: target=c2, control=c1
```

**Gate count per CCX:** 2 H + 4 T + 3 Tdg + 6 CX = 15 gates total, 7 T/Tdg gates

This is the standard decomposition from Nielsen & Chuang (Fig. 4.9), proven optimal for CNOT count (6 CNOTs is the lower bound, Shende & Markov 2008).

### CCX Decomposition Integration Points

When `toffoli_decompose` is active, there are two modes of decomposition:

**Mode A: Sequence generators (for hardcoded/cached sequences)**
- Sequence-building functions check `toffoli_decompose` flag
- Instead of calling `ccx()`, they call an inline helper that emits the 15-gate Clifford+T sequence
- New hardcoded sequence files generated by script contain pre-decomposed gate lists

**Mode B: Direct emit (for inline paths like `toffoli_mul_qq`)**
- The `emit_cMAJ_decomposed` / `emit_cUMA_decomposed` helpers in ToffoliMultiplication.c emit CCX gates via `add_gate()`
- When `toffoli_decompose` is on, they emit the 15-gate Clifford+T decomposition instead
- Helper function: `emit_ccx_decomposed(circ, target, ctrl1, ctrl2)` emits 15 gates via `add_gate()`

**The `toffoli_decompose` flag needs to be accessible both:**
1. From sequence generators (passed as parameter or global check)
2. From inline emitters (circuit_t field)

For sequences: the flag determines WHICH hardcoded sequence set to use (decomposed vs. standard).
For inline: the flag is checked at each CCX emission point.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CCX decomposition | Custom Clifford+T sequence | Standard 7T-6CX decomposition (Nielsen & Chuang) | Well-known optimal; any deviation introduces errors |
| MCX(3) decomposition | Ad-hoc 3-control elimination | AND-ancilla pattern from Phase 72-03 | Already proven correct in multiplication; compute-uncompute ensures ancilla cleanup |
| Recursive MCX(k>3) | Manual case-by-case decomposition | Recursive AND-ancilla reduction to k-1 controls | Generic approach handles comparison MCX(n) for arbitrary n |
| Hardcoded sequence generation | Manual C file construction | Python generation script (pattern from `generate_toffoli_seq.py`) | Avoids human error in 1000+ line C files; reproducible |

**Key insight:** The AND-ancilla decomposition pattern is already proven in `ToffoliMultiplication.c` lines 69-191. Phase 74 generalizes this exact pattern -- no new algorithm design needed.

## Common Pitfalls

### Pitfall 1: Ancilla Qubit Count Mismatch After MCX Decomposition
**What goes wrong:** MCX-decomposed cQQ sequences need additional AND-ancilla qubits beyond the original layout. If qubit_array mapping in hot_path_add.c / hot_path_mul.c doesn't account for the extra ancilla, the decomposition writes to wrong qubit indices.
**Why it happens:** The original cQQ_add sequences have qubit layout [0..bits-1]=a, [bits..2*bits-1]=b, [2*bits]=carry, [2*bits+1]=ext_ctrl. After MCX decomposition, an AND-ancilla qubit is needed at [2*bits+2] (or the inline decomposition must allocate it separately).
**How to avoid:** For sequence-based decomposition (hardcoded): include AND-ancilla in the static qubit layout. For inline decomposition: use `allocator_alloc()` to get AND-ancilla (already done in toffoli_mul_qq). Verify total qubit count in dispatch functions.
**Warning signs:** Gate purity tests show MCX gates remain; simulation produces wrong results for controlled additions.

### Pitfall 2: Breaking Existing Test Suite During Refactoring
**What goes wrong:** Moving code between files introduces header ordering issues, missing includes, broken static function visibility, or subtle initialization-order problems.
**Why it happens:** ToffoliAddition.c uses many static functions and local type definitions (bk_merge_t) that need to move together. If shared helpers like `alloc_sequence()` are called from multiple new files, linkage changes.
**How to avoid:** Do the refactoring as a pure behavior-preserving change FIRST. Run the full test suite after each file split. Use `static` keyword only for file-local functions; promote shared helpers to a header.
**Warning signs:** Compilation errors mentioning undefined functions; test regressions in unrelated tests.

### Pitfall 3: T/Tdg Gate Enum Conflicts
**What goes wrong:** Adding `T` to the enum causes macro name collisions or switch-statement fallthrough issues in circuit_output.c and gate.c.
**Why it happens:** `T` is a very common identifier in C (e.g., template parameters, test macros). The existing enum uses short names (X, Y, Z, H, P, M).
**How to avoid:** Use `T_GATE` and `TDG_GATE` enum names. Update ALL switch statements in circuit_output.c, gate.c, and circuit_stats.c to handle the new enum values (add cases; the default `other_gates` catch-all would hide the bug).
**Warning signs:** Compiler warnings about unhandled enum values in switch; QASM export outputs `// unknown gate` for T gates.

### Pitfall 4: Double Decomposition
**What goes wrong:** When both MCX decomposition and CCX->Clifford+T decomposition are active, the MCX decomposition produces CCX gates which then get decomposed again if the Clifford+T decomposition is also active. This is CORRECT behavior, but the sequence layer counts and total gate counts need to account for both levels.
**Why it happens:** The two decompositions compose multiplicatively: each MCX(3) becomes 3 CCX, each CCX becomes 15 Clifford+T gates, so each MCX(3) becomes 45 gates.
**How to avoid:** This is expected behavior, not a bug. But ensure hardcoded decomposed sequences are generated with BOTH decompositions applied (MCX -> CCX -> Clifford+T) rather than just one level.
**Warning signs:** None -- this is correct. But T-count tests need to check: original MCX(3) = 3 CCX * 7 T/Tdg = 21 T/Tdg per MCX(3).

### Pitfall 5: cQQ_add Hardcoded Sequences Still Contain MCX
**What goes wrong:** The existing hardcoded cQQ sequences in `toffoli_add_seq_N.c` files contain MCX(3) gates with dynamically allocated `large_control`. After Phase 74, these must still work OR be replaced with decomposed versions.
**Why it happens:** The `generate_toffoli_seq.py` script generates cMAJ/cUMA with MCX(3). The hardcoded sequences are loaded and cached.
**How to avoid:** Two options: (a) generate new decomposed hardcoded cQQ sequences that replace MCX with CCX, or (b) keep existing hardcoded sequences but ensure the dynamic generation path (bits > 8) also decomposes. Decision per CONTEXT.md: separate decomposed files.
**Warning signs:** Gate purity check fails for cQQ hardcoded widths 1-8.

### Pitfall 6: IntegerComparison MCX Must Also Be Decomposed
**What goes wrong:** IntegerComparison.c uses MCX for n-bit equality checks (CQ_equal_width, cCQ_equal_width). These are not Toffoli-arithmetic-only -- they run in both QFT and Toffoli modes. If MCX decomposition only applies to `fault_tolerant=True` mode, comparison MCX gates in QFT mode persist.
**Why it happens:** Per the user decision, MCX decomposition is "always-on when `fault_tolerant=True`". Comparison operations in QFT mode would still emit MCX. However, comparison MCX gates are n-controlled (not just 3-controlled), requiring recursive decomposition with (n-2) ancilla qubits.
**How to avoid:** For comparisons: MCX is only decomposed when `fault_tolerant=True`. QFT-mode comparisons keep MCX (MCX is a valid gate; only Toffoli mode needs gate purity). Ensure recursive decomposition handles arbitrary n.
**Warning signs:** Comparison tests fail with "unknown gate" or excessive ancilla consumption.

## Code Examples

### AND-ancilla MCX(3) Decomposition (proven pattern from Phase 72-03)

```c
// Source: ToffoliMultiplication.c lines 69-96
static void emit_cMAJ_decomposed(circuit_t *circ, int a, int b, int c,
                                  int ext_ctrl, int and_anc) {
    gate_t g;

    /* Step 1: CCX(target=b, ctrl1=c, ctrl2=ext_ctrl) */
    memset(&g, 0, sizeof(gate_t));
    ccx(&g, (qubit_t)b, (qubit_t)c, (qubit_t)ext_ctrl);
    add_gate(circ, &g);

    /* Step 2: CCX(target=a, ctrl1=c, ctrl2=ext_ctrl) */
    memset(&g, 0, sizeof(gate_t));
    ccx(&g, (qubit_t)a, (qubit_t)c, (qubit_t)ext_ctrl);
    add_gate(circ, &g);

    /* Step 3a: CCX(target=and_anc, ctrl1=a, ctrl2=b) -- compute AND */
    memset(&g, 0, sizeof(gate_t));
    ccx(&g, (qubit_t)and_anc, (qubit_t)a, (qubit_t)b);
    add_gate(circ, &g);

    /* Step 3b: CCX(target=c, ctrl1=and_anc, ctrl2=ext_ctrl) */
    memset(&g, 0, sizeof(gate_t));
    ccx(&g, (qubit_t)c, (qubit_t)and_anc, (qubit_t)ext_ctrl);
    add_gate(circ, &g);

    /* Step 3c: CCX(target=and_anc, ctrl1=a, ctrl2=b) -- uncompute AND */
    memset(&g, 0, sizeof(gate_t));
    ccx(&g, (qubit_t)and_anc, (qubit_t)a, (qubit_t)b);
    add_gate(circ, &g);
}
```

### CCX -> Clifford+T Decomposition Helper (NEW)

```c
// Standard exact Toffoli decomposition: 2H + 4T + 3Tdg + 6CX = 15 gates
// Source: Nielsen & Chuang Fig. 4.9, Shende & Markov arXiv:0803.2316
static void emit_ccx_clifford_t(circuit_t *circ, qubit_t target,
                                 qubit_t ctrl1, qubit_t ctrl2) {
    gate_t g;

    // H(target)
    memset(&g, 0, sizeof(gate_t));
    h(&g, target);
    add_gate(circ, &g);

    // CX(target, ctrl2)
    memset(&g, 0, sizeof(gate_t));
    cx(&g, target, ctrl2);
    add_gate(circ, &g);

    // Tdg(target)
    memset(&g, 0, sizeof(gate_t));
    tdg_gate(&g, target);
    add_gate(circ, &g);

    // CX(target, ctrl1)
    memset(&g, 0, sizeof(gate_t));
    cx(&g, target, ctrl1);
    add_gate(circ, &g);

    // T(target)
    memset(&g, 0, sizeof(gate_t));
    t_gate(&g, target);
    add_gate(circ, &g);

    // CX(target, ctrl2)
    memset(&g, 0, sizeof(gate_t));
    cx(&g, target, ctrl2);
    add_gate(circ, &g);

    // Tdg(target)
    memset(&g, 0, sizeof(gate_t));
    tdg_gate(&g, target);
    add_gate(circ, &g);

    // CX(target, ctrl1)
    memset(&g, 0, sizeof(gate_t));
    cx(&g, target, ctrl1);
    add_gate(circ, &g);

    // T(target), T(ctrl2)
    memset(&g, 0, sizeof(gate_t));
    t_gate(&g, target);
    add_gate(circ, &g);

    memset(&g, 0, sizeof(gate_t));
    t_gate(&g, ctrl2);
    add_gate(circ, &g);

    // H(target)
    memset(&g, 0, sizeof(gate_t));
    h(&g, target);
    add_gate(circ, &g);

    // CX(ctrl2, ctrl1)
    memset(&g, 0, sizeof(gate_t));
    cx(&g, ctrl2, ctrl1);
    add_gate(circ, &g);

    // T(ctrl1)
    memset(&g, 0, sizeof(gate_t));
    t_gate(&g, ctrl1);
    add_gate(circ, &g);

    // Tdg(ctrl2)
    memset(&g, 0, sizeof(gate_t));
    tdg_gate(&g, ctrl2);
    add_gate(circ, &g);

    // CX(ctrl2, ctrl1)
    memset(&g, 0, sizeof(gate_t));
    cx(&g, ctrl2, ctrl1);
    add_gate(circ, &g);
}
```

                                     qubit_t ctrl2) {
    // 15 gates, each in its own layer (sequential)
    h(&seq->seq[*layer][seq->gates_per_layer[*layer]++], target);
    (*layer)++;

    cx(&seq->seq[*layer][seq->gates_per_layer[*layer]++], target, ctrl2);
    (*layer)++;

    // ... etc (same 15-gate sequence as inline version)
    // Adds 15 layers per CCX instead of 1 layer
}

## Open Questions

1. **IntegerComparison MCX decomposition scope**
   - What we know: CQ_equal_width uses MCX(n) for n >= 3, cCQ_equal_width uses MCX(n+1). These require (n-2) AND-ancilla qubits for recursive decomposition.
   - What's unclear: Should these be decomposed when `fault_tolerant=True`? They use sequence-based emission into `seq->seq[layer]` rather than `add_gate(circ, &g)`. Inline decomposition requires knowing the circuit context, but these functions build standalone sequences.
   - Recommendation: For comparison functions, MCX decomposition requires allocating ancilla in the qubit layout upfront. This is more complex than the add/mul case. Consider decomposing comparison MCX as a separate task, or deferring comparison MCX decomposition to a later phase if it's not blocking.

2. **Hardcoded decomposed cQQ sequences vs inline decomposition**
   - What we know: User wants separate hardcoded decomposed files for fast dispatch. Generation script needs to emit CCX instead of MCX in cQQ sequences.
   - What's unclear: Whether the cCQ (controlled CQ) increment sequences also need decomposed variants (they contain MCX for bit=0 positions in cCQ_MAJ).
   - Recommendation: Generate decomposed cQQ hardcoded sequences first; cCQ can use dynamic generation with inline decomposition since they are value-dependent and already not cached.

3. **T/Tdg gate inverse relationship for `gates_are_inverse()` in gate.c**
   - What we know: `gates_are_inverse()` checks if two gates cancel. T and Tdg are inverses of each other.
   - What's unclear: Whether the optimizer currently uses this function for T/Tdg cancellation.
   - Recommendation: Add T_GATE/TDG_GATE cases to `gates_are_inverse()` so T followed by Tdg (or vice versa) on the same qubit is recognized as identity. Low effort, high correctness.
